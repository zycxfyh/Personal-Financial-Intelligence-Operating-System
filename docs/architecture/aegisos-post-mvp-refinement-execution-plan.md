# AegisOS Post-MVP Refinement Execution Plan

## Purpose

This document records the current post-MVP refinement order for AegisOS after the MVP gold path became real and delivery-gated.

The next phase is not broad platform expansion. It is owner cleanup, runtime-neutralization, operational maturity, advisory knowledge activation, and only then deeper console work.

## Execution Order

The correct order is:

1. `Boundary Purification`
2. `Runtime Identity Neutralization`
3. `Operational Maturity`
4. `Knowledge Activation`
5. `Console Maturation`

Each batch must prefer:

- owner cleanup over surface expansion
- read models over page-local inference
- supporting inspectors over premature full consoles
- `homepage / analyze / reviews` as the only first-class product pages in this phase

## Current Batch Reading

### 1. Boundary Purification

- continue finance pack extraction beyond analyze defaults
- keep finance semantics out of generic shell/page components
- continue reducing API compatibility-facade residue

### 2. Runtime Identity Neutralization

- keep runtime-neutral schema fields canonical
- continue reducing Hermes-specific identity wording and assumptions
- route runtime-facing behavior through adapter/runtime contracts

### 3. Operational Maturity

- stabilize scheduler trigger taxonomy
- strengthen monitoring/history summaries
- keep monitoring separate from business truth
- align runbooks and GitHub gate discipline with real signals

### 4. Knowledge Activation

- surface advisory knowledge through stable read paths
- keep candidate rules pre-policy and evidence-linked
- allow advisory consumption without mutating truth or bypassing governance

### 5. Console Maturation

- deepen the current MVP trio only
- add narrow inspectors for real data owners such as trace, governance detail, and monitoring history
- explicitly defer broad platform pages such as capability/runtime/pack/adapter consoles

## Exit Rule

A refinement batch is only complete when:

- owner boundaries are tighter than before
- no new finance leakage appears in generic UI
- no new Hermes identity leakage appears in app/API/web surfaces
- contract and snapshot changes are intentional
- the MVP gold path stays green
- docs and runbooks match the new code reality
